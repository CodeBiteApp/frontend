# 📋 CodeBite QA 분석 요약 및 조치 방안 보고서

본 문서는 퀴즈 시스템 동기화 분석, 상점/인벤토리 검증 로직, 그리고 로컬 배너 에셋 변조 대응책에 대해 항목별 1~2줄 분량으로 명확히 압축 정리한 보고서입니다.

## 1. 퀴즈 생성 및 답안 검증 동기화 문제점

- **A. 난수 엔진 불일치 (PRNG Engine Mismatch)**
  - **문제점**: 프론트엔드는 48비트 LCG(Java Random 규격)를 사용하는 반면 백엔드는 32비트 LCG(Numerical Recipes 규격)를 사용하여 동일 시드에서도 난수 시퀀스가 완전히 다름.
  - **수정 사항**: 백엔드의 `SeededRandom`을 프론트엔드에서 구현한 48비트 Java 표준 Random 스펙과 완전히 동일하게 고쳐 100% 동기화함.

- **B. 해시 기반 정렬 유실 (Lack of Sorting Guarantee)**
  - **문제점**: Shuffle 가공 전 입력 데이터(오답 풀 및 원천 데이터)의 사전식 정렬이 백엔드에서 누락되었고 순서 보장이 없는 `HashMap`을 사용하여 난수 매핑 순서가 꼬임.
  - **수정 사항**: 양측 모두 Shuffle 함수 직전에 입력 배열을 가나다/사전식 오름차순으로 명시적 정렬하고, `HashMap` 대신 `LinkedHashMap`이나 `List`를 사용함.

- **C. 비결정론적 데이터 생성 모델 (Algorithmic Asymmetry)**
  - **문제점**: 프론트엔드는 전체 문제 풀을 생성 후 시드 기반 균형 필터링(`selectBalancedQuestions`)을 태우나 백엔드는 단순 4문항 정적 수동 매핑을 진행하여 문제 세트 자체가 다름.
  - **수정 사항**: 퀴즈 생성 및 균형 추출 로직을 백엔드에 완벽하게 포팅하여 대칭구조로 구현하거나, 퀴즈 생성 API를 단일화하여 채점 대조군을 고정함.

- **💡 [추가 분석] 실행 테스트에서 채점/동기화 오류가 검출되지 않은 이유**
  - **원인 추측**: 현재 작성된 백엔드 단위 테스트(`ValidationServiceTest.java`)는 오답 및 인접 개념(Siblings)이 없는 단일 데이터 모형만 Mocking하여 **난수 연산이나 셔플을 아예 거치지 않는 `SHORT_ANSWER`(단답형) 케이스만 독립 검증**하고 있기 때문임. 이에 따라 버그가 존재하는 객관식, OX, 매칭형 문항 코드 분기를 타지 않아 불일치가 표면화되지 않았던 것으로 판단됨. 추가 확인 필요, 추후 예정.

## 2. 상점 및 인벤토리 백엔드 검증 문제점

- **A. 인벤토리 탐색 방식의 비효율성 (In-Memory Filtering)**
  - **문제점**: `equipItem` 시 사용자의 전체 보유 아이템을 DB에서 통째로 긁어온 뒤 Java Stream 메모리상에서 필터링하므로 인벤토리가 커질수록 지연 및 부하 유발.
  - **수정 사항**: `UserItemRepository`에 `findByUserIdAndItemId`를 신설하여 단일 아이템만 DB 레벨에서 즉각 정확히 특정하도록 수정함.

- **B. 단일 장착 슬롯 구조 (Slot Sharing)**
  - **문제점**: 배너(`BANNER`)와 프로필 아이콘(`ICON`)이 단 하나의 `equippedBannerId` 필드를 공유하여 장착 시 충돌할 여지가 존재함.
  - **수정 사항**: **단, `ICON` 아이템은 단순히 추가 계획 단계일 뿐 아직 확정되지 않은 사안**이므로, 향후 필요성 확정 시 전용 슬롯 분리 설계 검토.

- **C. 이중 구매 동시성 제어 부재 (Race Condition Vulnerability)**
  - **문제점**: `purchaseItem` 호출 시 보유 여부를 먼저 체크하고 `save`하는 과정에서 극히 짧은 찰나에 동시 요청(더블 클릭) 발생 시 동일 고유 아이템이 중복 구매될 위험이 있음.
  - **수정 사항**: 프론트엔드 구매 요청 트랜잭션 도중 **구매 버튼을 즉시 비활성화(Disabled)**하여 이중 클릭 및 중복 API 호출을 프론트 레벨에서 원천 차단하는 방향으로 수정함.

## 3. 치장성 로컬 배너 에셋 변조 위협 대응 방안

- **A. 개별 에셋 파일 체크섬 검증 방식 (Individual File Checksum Check)**
  - **문제점**: 클라이언트 기기(탈옥/루팅 환경) 로컬 파일을 덮어쓰는 행위는 프론트엔드 코드(JavaScript Bundle) 자체를 패칭하여 조건문 우회가 가능해 보안 실효성이 낮음.
  - **수정 사항**: 사용 방법이 단순하나 쉽게 무력화되므로 배포 오버헤드와 우회 가능성을 충분히 인지한 상태에서 소규모 임시 방편으로만 활용할 것을 권장.

- **B. 앱 바이너리 무결성 검증 방식 (App Binary Integrity & Attestation)**
  - **문제점**: 하드웨어 보안 및 OS 커널 검증 시스템 연동과 백엔드 연동 서버 구축이 필요하여 초기 아키텍처 연동 난이도가 높음.
  - **수정 사항**: **바이너리 무결성 검증(Play Integrity / App Attest) 구축을 우선하되, 개발 여력이 부족할 경우 최종 미구현 상태로 보류함.** (이미 서버 측에서 실시간 보유 확인 검증을 안전하게 차단하고 있어 타인 노출 악용 우려가 없음)
